# Enso Purity — Next.js frontend

Gemini-style dark/light UI for the tumor purity demo.

## Run locally

```bash
npm install
npm run dev
```

Open [http://localhost:3000](http://localhost:3000). Use the theme toggle in the header for dark/light mode.

## Data

- **Gallery (Case Explorer):** `public/gallery/gallery_summary.csv` and `public/gallery/interactive_<uuid>.html`.  
  On `npm run dev` / `npm run build`, `gallery/` is copied into `public/` so the app can serve these. Regenerate with `build_demo_gallery.py` (from repo root or `ml/` with env set), then copy outputs into `frontend/gallery/` (or run the script with `--out-dir frontend/gallery`).

- **Performance tab:**  
  - `public/data/statistical_tests.json` — from `python -m enso_purity_mil.statistical_tests ...`; copy `ml/runs/fold0/stats/statistical_tests.json` to `frontend/public/data/`.  
  - `public/data/per_cancer_stats.json` — from `python -m enso_purity_mil.per_cancer_statistical_tests ...`; copy `ml/runs/fold0/stats/per_cancer_stats.json` to `frontend/public/data/`.  
  - `public/data/scatter_mil_vs_ptn.png` — scatter plot (Enso MIL vs PTN vs genomic purity). Copy from VM: `scp ... ml/runs/fold0/stats/scatter_mil_vs_ptn.png frontend/public/data/`, or from the `demo_bundle` tar after running the bundle script on the VM.
  - `public/data/scatter_data.json` — same scatter data as JSON (genomic_purity, enso_mil, pathologist_ptn arrays). Generated by `statistical_tests.py` in the same run as the PNG; copy `ml/runs/fold0/stats/scatter_data.json` to `frontend/public/data/`. When present, the Performance tab shows an interactive theme-aware scatter with tooltips; otherwise it falls back to the static PNG.

## Generate all artifacts on the VM

On the GCP VM (bucket mounted, model and cache available), run once to generate stats, per-cancer stats, scatter data, and the full gallery (interactive heatmaps):

```bash
bash scripts/run_on_vm_generate_all.sh
```

Then copy `ml/runs/fold0/stats/*` to `frontend/public/data/` and `frontend/gallery/*` to `frontend/public/gallery/`. After that, visually review the heatmaps and add any slide with pen/marker to `data/exclude_markers.txt`, then re-run only the gallery step (see [docs/vm_generate_artifacts.md](../docs/vm_generate_artifacts.md)).

## One-per-cancer gallery

To build a gallery with one slide per cancer type (32–34 types), prefer “closer than pathologist”, and exclude marker slides:

1. Create a text file with one `file_uuid_original` or barcode per line for slides to exclude (e.g. marker slides).
2. Run:
   ```bash
   python -m enso_purity_mil.build_demo_gallery \
     --model-path ml/runs/fold0/best_model.pth \
     --manifest data/processed/wedge_mvp_dataset.xlsx \
     --h5-dir <path> --cache-dir <path> \
     --out-dir frontend/gallery \
     --one-per-cancer --err-limit 0.15 \
     --exclude-markers path/to/exclude_markers.txt
   ```
3. Copy `frontend/gallery/*` into `frontend/public/gallery/` if not using the predev/prebuild copy, or run `npm run dev` which copies `gallery/` to `public/`.

**Note:** If `build_demo_gallery` writes the CSV but the interactive viewer subprocess fails (e.g. H5 not found on gcsfuse or GDC thumbnail fetch fails), the Case Explorer will show the correct table and GDC links; the iframe will 404 until `interactive_<uuid>.html` files exist in `public/gallery/`. Run the viewer manually for each UUID when the H5 path is available, or fix the viewer’s H5/thumbnail paths on the VM and re-run the gallery build.

## Refreshing pathologist metadata (GDC API)

To refresh slide list and **pathologist (percent_tumor_nuclei)** metadata from the GDC API for all slides, from the **repository root**:

```bash
python backend/scripts/build_wedge_dataset.py
```

Defaults: `--slides-xlsx data/raw/slides_metadata_report(1).xlsx`, `--abs-tsv data/raw/TCGA_mastercalls.abs_tables_JSedit.fixed.txt`, `--out-dir data/processed`, `--reports-dir data/reports`. The script checks GDC file existence, fetches biospecimen and percent_tumor_nuclei from GDC, and writes `data/processed/wedge_mvp_dataset.xlsx`. Do not pass `--skip-gdc` if you want live API calls.

Optional wrapper (run from repo root):

```bash
./backend/scripts/fetch_gdc_metadata.sh
```
