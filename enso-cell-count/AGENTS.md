# AGENTS.md

## Cursor Cloud specific instructions

### Architecture

Single-product monorepo for tumor purity prediction from WSI tile embeddings. The **Python backend** (`backend/`) serves the API, `ml/` contains the MIL training and evaluation pipeline, `data/processed/` holds tracked canonical manifests used by training, and `docs/model_runs/` stores lightweight summaries for locked model campaigns. `frontend/` and `infra/` are placeholders.

### Running the backend

```bash
source backend/.venv/bin/activate
cd backend
uvicorn enso_purity.api.main:app --host 0.0.0.0 --port 8000 --reload
```

Endpoints: `GET /health`, `POST /predict_purity` (accepts `.h5` file upload).

### Running tests

```bash
source backend/.venv/bin/activate
cd backend && python -m pytest -q
```

All tests run offline (no network, no GPU). The model tests require **PyTorch** (CPU-only is fine), which is not listed in `requirements.txt` but is needed at runtime. The update script installs it.

### Linting

No linter is formally configured in the project. `ruff check backend/src backend/tests` works for quick checks. A few pre-existing lint warnings exist in the embedder script (`backend/src/enso_purity/embeddings/`) — these are upstream code and not blockers.

### Key caveats

- **PyTorch not in requirements.txt**: The model code and its tests import `torch`, but it is not declared in `backend/requirements.txt`. The update script installs CPU-only PyTorch separately.
- **Backend package install**: The backend must be installed in editable mode (`pip install -e backend`) so that `enso_purity` is importable. The update script handles this.
- **`python3.12-venv`**: The system Python may not have the `venv` module pre-installed. The update script installs the OS package if needed.
- **No database or Docker required**: Everything is file-based (H5, CSV, Parquet).
- **Swagger UI**: Available at `http://localhost:8000/docs` when the backend is running.
- **openpyxl / xlsxwriter**: Required for reading/writing `.xlsx` files (not in `requirements.txt`). The update script installs them.

### Data pipeline

The build_purity_dataset script (`backend/scripts/build_purity_dataset.py`) links TCGA slides to ABSOLUTE purity values. Run with `--skip-gdc` to avoid live GDC API calls (~2 min):

```bash
source backend/.venv/bin/activate
cd /workspace
python -m backend.scripts.build_purity_dataset --skip-gdc
```

Outputs go to `data/processed/` (xlsx) and `data/reports/` (plots + JSON stats). See `docs/dataset_linkage_spec.md` for details.

**Critical finding**: The slide barcode "portion" field does NOT correspond to the GDC biospecimen portion number. Slide portion `01` typically maps to biospecimen portion `11`. This means naive portion-level barcode matching yields only ~16% coverage. Use sample_vial-level matching (like SRTPMs) or GDC API for proper linkage.

### ML training pipeline (`ml/`)

Tests: `cd ml && python -m pytest tests/ -q` (42 tests, CPU-only, no H5 files needed).

#### GCP L4 training

Requires secrets: `GCP_SSH_HOST`, `GCP_SSH_USER`, `GCP_SSH_KEY_B64`.

```bash
# SSH setup (run once)
mkdir -p ~/.ssh
echo "$GCP_SSH_KEY_B64" | base64 -d > ~/.ssh/gcp_key
chmod 600 ~/.ssh/gcp_key
ssh -o StrictHostKeyChecking=no -i ~/.ssh/gcp_key "$GCP_SSH_USER@$GCP_SSH_HOST" hostname
```

Remote VM environment: Debian 11, NVIDIA L4 (23 GB), Python 3.11 via `~/miniconda3/envs/enso`. The embeddings bucket `embeddings-tcga-virchow` is mounted at `~/bucket_embeddings` via gcsfuse.

```bash
# On the remote VM
export PATH=~/miniconda3/envs/enso/bin:~/miniconda3/bin:$PATH
export HDF5_USE_FILE_LOCKING=FALSE
cd ~/enso_workspace

python -u -m enso_purity_mil.train_cli \
    --manifest data/processed/wedge_mvp_dataset.xlsx \
    --h5-dir ~/bucket_embeddings/embeddings_fp32 \
    --out-dir ml/runs \
    --device cuda --fold 0
```

**Key caveats for GCP training:**
- **`HDF5_USE_FILE_LOCKING=FALSE`** is mandatory — without it, h5py DataLoader workers crash over gcsfuse.
- **gcsfuse I/O is the bottleneck**, not GPU. Each epoch takes ~40 min with 4 workers. The dataset's smart I/O does sorted partial reads to minimize transfer.
- **No pre-caching** of tile counts in dataset init — this was removed because it sequentially opened ~12k H5 files over gcsfuse and took >1 hour.
- **Batch-level logging** every 20 batches in `training.py` so you can monitor progress during the long epochs.
- The GCS bucket must be mounted **without** a cache directory to ensure h5py partial reads don't trigger full-file downloads.

### Macrodissection workbench

The macrodissection workbench at `/macrodissection` is an AI-assisted ROI
adequacy guide for downstream molecular testing. The pathologist draws a
polygon on a digital H&E slide; the workbench computes the estimated tumor
purity (cellularity-weighted), total nuclei, tumor nuclei, area, and an
adequacy probability against configurable assay thresholds.

Code layout:

- `backend/src/enso_purity/macrodissection/` — Python package:
  - `inverse_cmap.py` / `grid_detect.py` / `build_artifacts.py` —
    pipeline that turns rendered PNG masks into per-tile JSON + Float32
    binary grids consumed by the frontend.
  - `roi.py` — Sutherland-Hodgman polygon-tile intersection, point
    estimates, deterministic Monte-Carlo sampler.
  - `adequacy.py` / `thresholds.py` — pass/borderline/fail labelling.
  - `candidates.py` — sliding-window candidate ROIs with rect NMS.
  - `storage.py` — append-only JSONL ROI store
    (`backend/.runtime/rois/case_{N}.jsonl`).
  - `report.py` / `router.py` — printable report payload + FastAPI router.

- `frontend/lib/macrodissection/` — TypeScript twins of the math:
  `polygon.ts`, `metrics.ts`, `adequacy.ts`, `thresholds.ts`,
  `colormaps.ts`, `smoothing.ts`, `overlay.ts`, `candidates.ts`,
  `tiles.ts`, `api.ts`, `static-cases.ts`, plus shared `types.ts`.

- `frontend/app/macrodissection/` — Next.js page + OpenSeadragon viewer +
  ROI/overlay components.

Run locally:

```bash
make backend-install
make backend-test                 # 57 pytest pass (incl. macrodissection)
make macrodissection-artifacts    # rebuild per-case JSON + bins from PNG masks
make macrodissection-serve        # uvicorn on :8000

make frontend-install
make frontend-test                # 40 vitest pass (TS math)
make frontend-build               # next build (static export)
make visual-check                 # Playwright -> docs/screenshots/*.png
```

Key caveats:

- **Tile-artifact generation is offline-buildable**: the CI VM does not need
  the trained model weights mounted. `build_artifacts` recovers per-tile
  predictions from the already-checked-in rendered PNG masks via
  inverse-colormap decoding. A future `--exact` flag will plug in raw
  scalar grids when they are available.
- **Identical math both sides**: the Python `roi.py` and the TS
  `metrics.ts` use the same Sutherland-Hodgman tile-clipping algorithm,
  the same Monte-Carlo formulation, and a deterministic seed derived from
  the polygon vertices, so the client preview and the authoritative
  server recompute always agree to ≤1e-3.
- **Visual smoothing is decoupled from metrics**: the overlay applies an
  alpha-aware Gaussian blur with a zoom-adaptive sigma, but the ROI
  metrics card always reads the raw tile grid.
- **Threshold profiles**: `humanitas_ngs` (default), `research`, and
  `strict_solid_tumor`. Per-call overrides are accepted at preview/save.
- **CORS / API**: the FastAPI app whitelists `localhost:3000` and
  `localhost:3001` (Next dev server) by default; extra origins can be
  supplied via `ENSO_API_CORS_ORIGINS`.
