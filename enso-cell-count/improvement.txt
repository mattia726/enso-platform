## What I verified in the repo

The implementation does match the architecture we discussed. `model.py` has the 2560-dimensional Virchow input, projection to 512, learned 3×3 positional embedding, masked center-query multi-head attention, metadata MLP, residual MLP trunk, density head, dispersion head, ordinal head, quantile head, and quality head. It also computes:

```python
density = softplus(...)
mu = density * exposure_mm2
alpha = softplus(...)
theta = 1 / alpha
nb_logits = log(mu) - log(theta)
```

That is the right structure for the NB2-style count model. 

The composite loss is also implemented in the expected way: Negative Binomial NLL, SmoothL1 on `log1p(mu)`, ordinal BCE, quantile pinball loss on log-counts, and optional quality loss. Teacher confidence weighting is applied to the count-related components. 

The training loop uses AdamW with decay exclusions for biases/norms, gradient clipping, AMP support, per-epoch metric accumulation, ReduceLROnPlateau, early stopping, and checkpointing. 

The train CLI saves `best_model.pth` based on **validation loss**, not raw MAE. That means your current “best model” is likely epoch 5, not epoch 9, even though epoch 9 is best for raw count MAE. 

The dataset implementation samples tiles per slide, dynamically gathers 3×3 neighbor embeddings from the H5, constructs metadata, exposure, teacher count, teacher confidence, quality target, and count bin, then flattens slide-sampled items into tile batches.   

The folds are case-level, and the code explicitly asserts no case leakage between train, validation, and test. That is very good and removes the biggest easy source of inflated validation performance. 

The label pipeline reads Pan-Cancer-Nuclei-Seg DICOM ANN polygons, computes centroids, counts nuclei centroids into non-overlapping embedding tiles, and writes one Parquet label table per slide.   

There are unit tests for centroid counting, ordinal bins, model output shapes, quantile ordering, valid center tile enforcement, composite loss finiteness, 3×3 neighbor gathering, one training epoch, inference, DICOM primitive index handling, and Pan-Cancer metadata parsing.   

So the implementation is not a random toy model. It is a real, sensible implementation of the architecture.

## Is it underfitting?

No.

Evidence:

```text
Epoch 1 val MAE: 24.39
Epoch 3 val MAE: 8.51
Epoch 5 val MAE: 7.04
Epoch 9 val MAE: 6.78
```

The model learned most of the usable signal within the first 3–5 epochs. An underfitting model would usually have both high train MAE and high validation MAE, and it would continue improving if you gave it more epochs. Here, it reached a low validation error quickly and then plateaued.

The architecture has enough capacity: it is using a rich Virchow embedding, 3×3 context attention, metadata, and residual nonlinear heads. There is no evidence from the table that capacity is too small.

## Is it overfitting?

Not in the classic sense.

Classic overfitting would look like:

```text
train MAE keeps falling
val MAE keeps rising
train/val gap widens strongly
```

That is not what your table shows. In fact, from epoch 10 onward, train MAE is often **higher** than validation MAE:

```text
epoch 12: train MAE 9.08, val MAE 7.45
epoch 14: train MAE 9.47, val MAE 7.59
epoch 16: train MAE 9.11, val MAE 7.19
```

That does **not** mean validation is magically easier in a suspicious way by itself. The repo explains why this can happen:

The train dataset uses `balanced_bins` sampling by default, while validation uses uniform sampling.  

Balanced-bin sampling likely over-represents harder high-count bins during training, while uniform validation reflects the natural tile distribution. Also, train metrics are measured with dropout active, while validation runs with dropout disabled. So train/validation MAE are not perfectly apples-to-apples.

The more accurate diagnosis is:

```text
Not underfitting.
Not severe overfitting.
Plateaued after epoch 5–9.
Mild overtraining or noisy validation after epoch 9.
```

If raw count MAE is the product KPI, epoch 9 is currently your best. If calibrated uncertainty/composite behavior is the KPI, epoch 5 may be better.

## The biggest implementation issue: validation sampling changes every epoch

This is the main thing I would fix.

In `train_cli.py`, each epoch does:

```python
train_ds.set_epoch(epoch)
val_ds.set_epoch(epoch)
```

The dataset RNG depends on `self.epoch`, so validation samples a different subset of tiles each epoch.  

That means your validation MAE/loss are partly measuring a changing validation subset, not just the changing model. This can explain why validation loss bounces around while MAE stays in the same narrow range.

For model selection, validation should be fixed or exhaustive.

I would change this:

```python
train_ds.set_epoch(epoch)
val_ds.set_epoch(epoch)
```

to this:

```python
train_ds.set_epoch(epoch)

# Keep validation deterministic across epochs.
# This makes checkpoint selection much less noisy.
val_ds.set_epoch(0)
```

Even better, for final selection, run validation on all tiles by setting:

```bash
--eval-tiles-per-slide 0
```

because `_sample_indices` returns all rows when `n <= 0`. 

## Second issue: checkpointing is optimized for loss, not the doctor-facing metric

Right now `best_model.pth` is saved only when `val_loss` improves. 

That is why epoch 5 wins as the saved best model:

```text
epoch 5 val loss = 3.6709
epoch 9 val MAE  = 6.78 but val loss = 4.3335
```

This discrepancy is expected because the loss is not just MAE. It includes NB calibration, log-count SmoothL1, ordinal BCE, quantile loss, and quality loss. 

For this product, I would save at least three checkpoints:

```text
best_by_val_loss.pth
best_by_val_mae_count.pth
best_by_val_mae_log1p.pth or best_by_within_factor2.pth
```

The doctor-facing point estimate should probably use `best_by_val_mae_count` or `best_by_mae_log1p`, while the uncertainty/calibration model should be judged by interval coverage and NB loss.

A minimal code change:

```python
best_val_loss = float("inf")
best_val_mae = float("inf")
best_val_log_mae = float("inf")

...

if val_loss < best_val_loss:
    best_val_loss = val_loss
    save_checkpoint(fold_dir / "best_by_loss.pth", ...)

if val_metrics["mae_count"] < best_val_mae:
    best_val_mae = float(val_metrics["mae_count"])
    save_checkpoint(fold_dir / "best_by_mae_count.pth", ...)

if val_metrics["mae_log1p"] < best_val_log_mae:
    best_val_log_mae = float(val_metrics["mae_log1p"])
    save_checkpoint(fold_dir / "best_by_mae_log1p.pth", ...)
```

I would also log `val_mae_count` directly in the console message. The metric is already computed in `metrics.py`; it is just not part of the main logger line. 

## Third issue: quality head is probably not meaningful yet

The architecture includes a quality/artifact head, but the current label builder appears to set:

```python
quality_flags = "pan_cancer_ann"
```

for all generated labels. 

Then `_quality_targets` maps missing/neutral quality flags to class 0 unless the string contains terms like background, empty, artifact, blur, necrosis, or fold. 

So in the current Pan-Cancer labels, the quality head is probably being trained to say “good” for almost everything. That means the quality loss is not yet clinically meaningful. It may not hurt much, but it can distort `val_loss` and partially explain why best loss and best MAE disagree.

For the current training runs, I would set:

```bash
--quality-weight 0.0
```

until you have real quality labels, or derive them from an actual tissue/background/artifact detector.

## Fourth issue: exposure normalization exists, but tissue fraction is currently constant

The model is correctly designed as:

```text
predicted density × exposure_mm2 = predicted count
```

But the current label generation sets:

```python
tissue_fraction = 1.0
exposure_mm2 = tile_area
```

for every tile. 

Inference also builds metadata with tissue fraction 1.0 and exposure equal to full tile area. 

So right now, the model is effectively learning **count per full tile**, not true tissue-adjusted count. That is okay for a first model, especially if tiles are mostly tissue-containing. But for the doctor selection use case, partial tissue/background tiles matter.

I would add a tissue-mask-derived field:

```text
tissue_fraction
exposure_mm2 = tile_area_mm2 × tissue_fraction
```

Then the model can separate:

```text
low count because tissue is sparse/background
vs.
low count because tissue is present but hypocellular
```

That distinction matters clinically.

## 1. Quality: it had a quality head/loss, but not meaningful quality supervision

The model architecture includes a `quality_head` with 3 quality classes, and the default CLI has:

```text
--num-quality-classes = 3
--quality-weight = 0.20
```

So unless you explicitly launched training with:

```bash
--quality-weight 0.0
```

the composite `train_loss` and `val_loss` did include a **quality loss component**. The default loss weights include `quality: 0.20`, and `train_cli.py` passes that into `CellularityLossWeights`.  

But the quality labels themselves were basically not real QC labels.

The label builder sets:

```python
"quality_flags": "pan_cancer_ann"
```

for every tile. It does **not** create real labels like background, blur, fold, necrosis, artifact, etc. 

Then the dataset code does this:

```python
if "quality_target" in labels.columns:
    use labels["quality_target"]
elif "quality_flags" not in labels.columns:
    return zeros
else:
    map "background|empty" -> 1
    map "artifact|blur|necrosis|fold" -> 2
    otherwise remain 0
```

Since `"pan_cancer_ann"` does not contain those keywords, almost/all tiles become:

```text
quality_target = 0
```

which means “good/default tissue.” 

So the answer is:

```text
The model was using a quality loss term, but it was not using real quality metrics or real quality labels.
```

It was mostly learning:

```text
everything is quality class 0
```

That can still contribute to `val_loss`, but it is not clinically meaningful and does not tell you whether tiles are blurry, folded, necrotic, background, etc.

Also, the reported metrics did **not** include quality accuracy, quality F1, artifact recall, etc. The metrics accumulator tracks count MAE, log MAE, factor-2 accuracy, bin accuracy, interval coverage, mean true/predicted count, and Spearman. It does not compute quality metrics. 

So the visible table’s MAE values were pure count metrics. The `Val Loss`, however, likely included the degenerate quality-loss component if you used the default `--quality-weight 0.20`.

## 2. Tissue fraction: it was always 1.0

The current label builder sets:

```python
tissue_fraction = np.ones(len(counts), dtype=np.float32)
exposure_mm2 = tile_area
```

for every tile. 

The dataset then reads those fields into metadata:

```python
tissue_fraction = labels.get("tissue_fraction", 1.0)
tile_area = labels["tile_area_mm2"]
exposure = labels["exposure_mm2"]
```

and returns normalized metadata including tissue fraction and exposure. 

The model does use exposure mathematically:

```python
density = softplus(density_head)
mu = density * exposure_mm2
```

So architecturally it supports exposure/tissue-area correction. 

But because `exposure_mm2` was always the full tile area, the current run was effectively doing:

```text
predicted count = predicted density × full tile area
```

not:

```text
predicted count = predicted density × actual tissue-covered tile area
```

Inference does the same thing: for unlabeled H5 inference, it creates:

```python
tissue_fraction = ones
exposure_mm2 = tile_area
```

so it also assumes every tile has full tissue exposure. 

So the answer is:

```text
The model was using tissue_fraction as an input field, but the actual value was always 1.0.
It was not using a real tissue mask or real partial-tissue fraction.
```

It did **not** learn:
